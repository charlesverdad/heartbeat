# HB-Pulse Implementation Guide

Deploy Pulse (members-only knowledgebase PWA) to the existing VM at `pulse.heartbeatchurch.com.au`.

## Architecture Overview

```
Internet → Cloudflare (SSL/DNS) → Cloudflare Tunnel → VM (localhost:3001)
                                                        ├── pulse (SvelteKit app, port 3001)
                                                        ├── pulse-db (PostgreSQL 16)
                                                        └── cloudflared (tunnel connector)
```

Coexists with the living-life quiz on the same VM. Pulse uses port **3001** (quiz uses 3000).

## Prerequisites

### Secrets to Generate/Obtain Before Starting

| Secret | How to generate |
|--------|----------------|
| `session-secret` | `openssl rand -hex 32` |
| `db-password` | `openssl rand -hex 16` |
| `vapid-public-key` | `npx web-push generate-vapid-keys` (save both keys) |
| `vapid-private-key` | (generated with above) |
| `google-client-id` | Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client ID |
| `google-client-secret` | (generated with above) |
| `smtp-pass` | Google Workspace → App Passwords (for `noreply@heartbeatchurch.com.au`) |
| `cloudflare-tunnel-token` | Auto-generated by Terraform |

### Google OAuth Setup

1. Go to [Google Cloud Console](https://console.cloud.google.com/) → APIs & Services → Credentials
2. Create an OAuth 2.0 Client ID (Web application)
3. Set **Authorized redirect URI**: `https://pulse.heartbeatchurch.com.au/auth/callback`
4. Note the Client ID and Client Secret

### VAPID Keys

```bash
npx web-push generate-vapid-keys
```

Save both the public and private keys — they go into Key Vault.

## Step 1: Terraform — Infrastructure Provisioning

Create `terraform/vm-hb-pulse/` following the same pattern as `terraform/vm-living-life-quiz/`.

### 1.1 Files to Create

```
terraform/vm-hb-pulse/
├── backend.tf
├── main.tf
├── variables.tf
├── outputs.tf
├── providers.tf
└── prod.tfvars
```

### 1.2 `backend.tf`

State stored in the shared Azure Storage backend:
- Resource group: `rg-terraform-state-backend`
- Storage account: `terraforminfratfstate`
- Container: `tfstate`
- Key: `vm-hb-pulse/terraform.tfstate`

### 1.3 `providers.tf`

Same as living-life-quiz: `azurerm ~> 3.0`, `cloudflare ~> 4.0`, `random ~> 3.0`.

### 1.4 `main.tf` — Resources to Provision

Reference the **existing** VM's resource group and managed identity (same VM as living-life quiz):

|----------|-------------|---------|

| `azurerm_role_assignment` (x2) | — | VM identity → "Key Vault Secrets User"; current user → "Key Vault Administrator" |
| **Secrets (8):** | | |
| `azurerm_key_vault_secret` | `session-secret` | Session signing key |
| `azurerm_key_vault_secret` | `db-password` | PostgreSQL password |
| `azurerm_key_vault_secret` | `google-client-id` | OAuth client ID |
| `azurerm_key_vault_secret` | `google-client-secret` | OAuth client secret |
| `azurerm_key_vault_secret` | `vapid-public-key` | Web Push public key |
| `azurerm_key_vault_secret` | `vapid-private-key` | Web Push private key |
| `azurerm_key_vault_secret` | `smtp-pass` | Google App Password for email |
| `azurerm_key_vault_secret` | `cloudflare-tunnel-token` | Auto-generated tunnel token |

### 1.7 Tunnel Config — Important Difference

The tunnel ingress must point to **port 3001** (not 3000, which is used by living-life quiz):

```hcl
ingress_rule {
  hostname = var.domain_name
  service  = "http://localhost:3001"
  origin_request {
    http_host_header = var.domain_name
  }
}
```

### 1.8 Apply

```bash
cd terraform/vm-hb-pulse
terraform init
terraform plan -var-file=prod.tfvars
terraform apply -var-file=prod.tfvars
```

Note the Key Vault name from the output — needed for `mount-secrets.sh`.

## Step 2: VM Deployment Manifests

Create `manifests/hb-pulse/` with three files:

### 2.1 `mount-secrets.sh`

Same pattern as living-life quiz. Fetches secrets from the **Pulse** Key Vault using the VM's managed identity.

Required secrets to fetch:
```bash
REQUIRED_SECRETS=(
    "session-secret"
    "db-password"
    "google-client-id"
    "google-client-secret"
    "vapid-public-key"
    "vapid-private-key"
    "smtp-pass"
    "cloudflare-tunnel-token"
)
```

Mount to: `/secrets/hb-pulse/` (separate from `/secrets/` used by living-life quiz).

Run as: `sudo KEY_VAULT_NAME=<vault-name> ./mount-secrets.sh`

### 2.2 `docker-compose.yml`

```yaml
version: '3.8'

services:
  pulse-db:
    image: postgres:16-alpine
    container_name: pulse-db
    restart: unless-stopped
    volumes:
      - /data/hb-pulse/postgres:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: pulse
      POSTGRES_USER: pulse
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    healthcheck:
      test: ['CMD-SHELL', 'pg_isready -U pulse']
      interval: 5s
      timeout: 5s
      retries: 5
    networks:
      - pulse-network

  pulse:
    image: pulse:latest
    container_name: pulse
    restart: unless-stopped
    ports:
      - "3001:3000"
    depends_on:
      pulse-db:
        condition: service_healthy
    volumes:
      - /data/hb-pulse/uploads:/data/uploads
    environment:
      - DATABASE_URL=postgresql://pulse:${DB_PASSWORD}@pulse-db:5432/pulse
      - SESSION_SECRET=${SESSION_SECRET}
      - GOOGLE_CLIENT_ID=${GOOGLE_CLIENT_ID}
      - GOOGLE_CLIENT_SECRET=${GOOGLE_CLIENT_SECRET}
      - BASE_URL=https://pulse.heartbeatchurch.com.au
      - ADMIN_EMAIL=${ADMIN_EMAIL}
      - PORT=3000
      - NODE_ENV=production
      # Email (Google SMTP)
      - SMTP_HOST=smtp.gmail.com
      - SMTP_PORT=587
      - SMTP_USER=noreply@heartbeatchurch.com.au
      - SMTP_PASS=${SMTP_PASS}
      - SMTP_FROM=noreply@heartbeatchurch.com.au
      - SMTP_FROM_NAME=Pulse — Heartbeat Church
      # Push notifications
      - VAPID_PUBLIC_KEY=${VAPID_PUBLIC_KEY}
      - VAPID_PRIVATE_KEY=${VAPID_PRIVATE_KEY}
      - VAPID_SUBJECT=mailto:admin@heartbeatchurch.com.au
    healthcheck:
      test: ['CMD', 'wget', '-qO-', 'http://localhost:3000/health']
      interval: 5s
      timeout: 5s
      retries: 10
      start_period: 10s
    networks:
      - pulse-network

  cloudflared-pulse:
    image: cloudflare/cloudflared:latest
    container_name: cloudflared-pulse
    restart: unless-stopped
    network_mode: host
    environment:
      - TUNNEL_TOKEN=${TUNNEL_TOKEN}
    command: ["tunnel", "run", "--token", "${TUNNEL_TOKEN}"]
    depends_on:
      - pulse

networks:
  pulse-network:
    driver: bridge
```

**Key differences from living-life quiz:**
- Port mapping: `3001:3000` (host 3001 → container 3000) to avoid conflict
- Separate `pulse-network` (isolated from quiz containers)
- PostgreSQL as a sidecar (quiz uses SQLite)
- Persistent data at `/data/hb-pulse/` (quiz uses `/data/living-life-quiz/`)
- Container names prefixed with `pulse-` to avoid conflicts

### 2.3 `deploy.sh`

Reads secrets from `/secrets/hb-pulse/` and exports them as environment variables:

```bash
#!/bin/bash
set -euo pipefail

# Check secrets exist
SECRETS_DIR="/secrets/hb-pulse"
if [[ ! -d "$SECRETS_DIR" ]]; then
    echo "ERROR: Run mount-secrets.sh first"
    exit 1
fi

# Export secrets from Key Vault files
export DB_PASSWORD=$(sudo cat $SECRETS_DIR/db-password)
export SESSION_SECRET=$(sudo cat $SECRETS_DIR/session-secret)
export GOOGLE_CLIENT_ID=$(sudo cat $SECRETS_DIR/google-client-id)
export GOOGLE_CLIENT_SECRET=$(sudo cat $SECRETS_DIR/google-client-secret)
export VAPID_PUBLIC_KEY=$(sudo cat $SECRETS_DIR/vapid-public-key)
export VAPID_PRIVATE_KEY=$(sudo cat $SECRETS_DIR/vapid-private-key)
export SMTP_PASS=$(sudo cat $SECRETS_DIR/smtp-pass)
export TUNNEL_TOKEN=$(sudo cat $SECRETS_DIR/cloudflare-tunnel-token)
export ADMIN_EMAIL="charles@heartbeatchurch.com.au"  # First admin

# Create data directories
sudo mkdir -p /data/hb-pulse/postgres
sudo mkdir -p /data/hb-pulse/uploads

# Deploy
docker compose down || true
docker compose up -d

# Verify
sleep 5
docker compose ps
curl -f http://localhost:3001/health && echo "Pulse is healthy"
```

## Step 4: Post-Deployment

### 4.1 First Admin Login

1. Navigate to `https://pulse.heartbeatchurch.com.au`
2. Log in with Google using the `ADMIN_EMAIL` account
3. That account will be automatically granted admin role

### 4.2 Backup Considerations

The existing VM data disk backup (daily, 60-day retention via `terraform/vm/backup.tf`) covers `/data/` which includes both:
- `/data/living-life-quiz/` (SQLite)
- `/data/hb-pulse/postgres/` (PostgreSQL data)
- `/data/hb-pulse/uploads/` (uploaded files)

**No additional backup infrastructure is needed** — the existing Azure Backup Vault already protects the data disk.

### 4.3 VM Resource Constraints

Current VM is `Standard_B1ms` (1 vCPU, 2 GB RAM). With both apps running:
- living-life-quiz: ~50-100 MB RAM (Node.js + SQLite)
- Pulse: ~200-400 MB RAM (Node.js + PostgreSQL)
- Two cloudflared instances: ~50 MB each
- **Total estimated: ~400-600 MB** — fits within 2 GB but monitor with `docker stats`

If the VM becomes resource-constrained, consider upgrading to `Standard_B2s` (2 vCPU, 4 GB RAM) via `terraform/vm/prod.tfvars`.
